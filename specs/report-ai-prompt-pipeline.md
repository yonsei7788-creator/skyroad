## 1. 프롬프트 아키텍처 총괄

### 1.1 설계 원칙

1. **섹션별 분할 호출** -- 21개 섹션을 개별 API 호출로 분리 (이유: 출력 품질 제어, 재시도 격리, 토큰 제한 관리)
2. **구조화된 출력 강제** -- Gemini의 JSON mode + response_schema로 출력 형식을 100% 보장
3. **원문 인용 기반 평가** -- 할루시네이션 방지를 위해 모든 평가는 생기부 원문 인용을 수반
4. **플랜별 프롬프트 분기** -- 동일 섹션이라도 Lite/Standard/Premium별로 다른 지시문과 출력 스키마 적용
5. **4대 역량 프레임워크 관통** -- 시스템 프롬프트에 역량 프레임워크(학업/진로/공동체/발전가능성)를 내재화하여 모든 섹션에서 일관된 평가 축 사용

### 1.2 호출 전략: 섹션별 분할 생성

**선택 근거**: 한 번에 전체 리포트(22~28P)를 생성하면 (1) 출력 토큰 제한 초과 위험, (2) 후반부 품질 저하, (3) 일부 섹션 실패 시 전체 재생성 필요. 섹션 그룹 호출(2~3개씩 묶기)은 출력 스키마 복잡도 증가로 파싱 에러 위험.

```
[호출 구조]

Phase 1: 전처리 (코드 로직, AI 호출 없음)
  └─ 생기부 원문 파싱 + 정규화 + 프로필 데이터 구조화
     + 워드클라우드 전처리 (키워드 빈도 계산)
     + 출결 데이터 정규화
     + 기록 분량 계산 (max capacity vs actual)

Phase 2: 기반 분석 (AI 호출 3건, 병렬)
  ├─ Call A: 역량 추출 (내부용, 사용자 미노출)
  ├─ Call B: 성적 맥락 분석 (내부용, 사용자 미노출)
  └─ Call C: 학생 유형 분류 (내부용, studentProfile + competencyScore 기반)

Phase 3: 섹션 생성 (AI 호출 ~18건, 의존성 기반 순차/병렬)
  ├─ [Group 1 - 의존: A,B,C]
  │   ├─ Call 1: studentProfile (학생 프로필)
  │   ├─ Call 2: competencyScore (역량 정량 스코어)
  │   └─ Call 3: diagnostic (종합 진단)
  ├─ [Group 2 - 의존: A,B]
  │   ├─ Call 4: competencyEvaluation (역량별 종합 평가)
  │   ├─ Call 5: academicAnalysis (성적 분석)
  │   ├─ Call 6: courseAlignment (권장과목 이수 분석)
  │   ├─ Call 7: attendanceAnalysis (출결 분석, 코드 전처리 + AI 해석)
  │   └─ Call 8: activityAnalysis (창체 활동 분석)
  ├─ [Group 3 - 의존: A]
  │   ├─ Call 9: subjectAnalysis (교과 세특 분석)
  │   ├─ Call 10: behaviorAnalysis (행동특성 분석, Standard+)
  │   └─ Call 11: overallAssessment (기록 충실도 종합, Standard+)
  ├─ [Group 4 - 의존: Group 2,3]
  │   ├─ Call 12: weaknessAnalysis (부족한 부분 + 보완 전략)
  │   ├─ Call 13: admissionPrediction (합격 예측)
  │   ├─ Call 14: topicRecommendation (세특 주제 추천)
  │   └─ Call 15: interviewPrep (예상 면접 질문, Standard+)
  ├─ [Group 5 - 의존: Group 4]
  │   ├─ Call 16: admissionStrategy (입시 전략 + 대학 추천)
  │   ├─ Call 17: storyAnalysis (스토리 구조 분석, Standard+)
  │   └─ Call 18: actionRoadmap (실행 로드맵, Standard+)
  └─ [Group 6 - 부록, 의존: A]
      ├─ Call 19: bookRecommendation (추천 도서, Standard+)
      ├─ Call 20: majorExploration (AI 전공 추천, Standard+)
      └─ wordCloud (코드 전처리, AI 호출 없음)

Phase 4: 후처리 (코드 로직, AI 호출 없음)
  └─ JSON 스키마 검증 + 섹션 조합 + 저장
```

**병렬 실행 그룹**:

- Phase 2: Call A + Call B + Call C (병렬)
- Group 1: Call 1 + Call 2 + Call 3 (병렬, Phase 2 완료 후)
- Group 2: Call 4 + Call 5 + Call 6 + Call 7 + Call 8 (병렬, Phase 2 완료 후)
- Group 3: Call 9 + Call 10 + Call 11 (병렬, Phase 2 완료 후)
- Group 4: Call 12 + Call 13 + Call 14 + Call 15 (병렬, Group 2/3 완료 후)
- Group 5: Call 16 + Call 17 + Call 18 (병렬, Group 4 완료 후)
- Group 6: Call 19 + Call 20 (병렬, Phase 2 완료 후, 독립)

### 1.3 구조화된 출력 강제 방법

Gemini 2.5 Pro의 **Structured Output** (JSON mode with response_schema) 사용:

```typescript
// 호출 예시 (Gemini API)
const result = await model.generateContent({
  contents: [{ role: "user", parts: [{ text: prompt }] }],
  generationConfig: {
    responseMimeType: "application/json",
    responseSchema: sectionSchema, // 섹션별 JSON Schema
    temperature: 0.3, // 평가 일관성을 위해 낮은 temperature
  },
  systemInstruction: systemPrompt,
});
```

**핵심**: `responseSchema`를 섹션별로 정의하여 Gemini가 해당 스키마를 100% 준수하는 JSON만 출력하도록 강제. 자유 텍스트 필드(평가 코멘트 등)의 품질은 프롬프트로 제어.

## 2. 시스템 프롬프트 설계

### 2.1 공통 시스템 프롬프트 (모든 호출에 적용)

```
당신은 대한민국 입시 전문 AI 분석가입니다. 대학 입학사정관의 평가 관점과 입시 컨설턴트의 전략적 시각을 겸비하고 있습니다.

## 역할
- 학생의 학교생활기록부(생기부)를 분석하여 입시에 도움이 되는 전문적인 평가와 전략을 제공합니다.
- 평가는 반드시 생기부 원문에 근거해야 하며, 원문에 없는 내용을 추측하거나 만들어내지 않습니다.
- 리포트는 3파트 + 부록, 총 21개 섹션으로 구성됩니다:
  - Part 1 진단: studentProfile, competencyScore, admissionPrediction, diagnostic
  - Part 2 분석: competencyEvaluation, academicAnalysis, courseAlignment, attendanceAnalysis, activityAnalysis, subjectAnalysis, behaviorAnalysis, overallAssessment
  - Part 3 전략: weaknessAnalysis, topicRecommendation, interviewPrep, admissionStrategy, storyAnalysis, actionRoadmap
  - 부록: bookRecommendation, majorExploration, wordCloud

## 4대 평가 역량 프레임워크

모든 평가는 다음 4대 역량을 기준으로 수행합니다:

### 학업역량
- 학업성취도: 내신 등급, 등급 추이, 과목 간 편차, 수강자수/표준편차 대비 실질 위치
- 학업태도: 수업 참여도, 자기주도적 학습, 성실성 (세특 서술에서 추출)
- 탐구력: 지적 호기심, 탐구의 깊이/확장성, 꼬리물기 탐구 여부, 과정 중심 서술

### 진로역량
- 전공 관련 교과 이수 노력: 핵심/일반 권장과목 이수 여부, 공동교육과정, 진로선택과목
- 전공 관련 교과 성취도: 전공 관련 과목의 성적이 전체 대비 어떤지
- 진로 탐색 활동 및 경험: 진로 방향 일관성, 학년별 심화도, 진로 관련 구체적 활동

> **주의**: 단순히 직업을 탐구한 경우 해당 전공분야 관련 역량이 드러나는 것이 아닙니다. 학생의 진로를 제시할 수 있지만, 단순 직업 탐구가 진로역량에 긍정적 영향이 있는 것은 아니므로 이 방향으로 평가해야 합니다.

### 공동체역량
- 나눔과 배려: 친구 학습 도움, 타인 의견 존중, 배려 사례
- 소통 및 협업: 모둠 활동, 토론 참여, 발표 역량, 팀 프로젝트
- 리더십: 임원 활동, 모둠장/부장 역할, 주도적 기획
- 성실성과 규칙준수: 출결, 역할 이행, 과제 수행 성실도

### 발전가능성
- 자기주도성: 스스로 질문을 만들고 학습을 확장하는 경험, 교사 지도 없이 추가 탐구 수행
- 경험의 다양성: 다양한 영역의 활동 경험, 교과 내/외 경험의 폭
- 성장 과정: 학년별 활동의 점진적 심화, 시행착오와 한계 인식, 보완 시도의 흔적
- 창의적 문제해결: 기존과 다른 접근, 독창적 아이디어, 문제 정의-해결-확장의 사고 흐름

> **핵심 질문**: "이 학생이 대학에 와서도 계속 성장할 수 있는가?"
> 성공 경험뿐 아니라 시행착오, 한계 인식, 보완 시도가 오히려 긍정적 평가 요소

## 평가 원칙

1. **과정 중심 평가**: 결과가 아닌 탐구 과정(접근 방식, 사고 과정, 변화)을 중심으로 평가
2. **구체성 기반 판단**: "심도 있게 탐구함" 같은 추상적 표현보다 실제 내용의 구체성으로 판단
3. **맥락적 성적 해석**: 등급만이 아닌 원점수-평균-표준편차-수강자수 종합 분석
4. **연결성 중시**: 개별 활동이 아닌 활동 간 연결, 학년 간 심화를 종합 판단
5. **학년별 차등 기대**: 1학년은 탐색기로 관대, 2학년은 구체화 기대, 3학년은 심화 기대
6. **형식적 표현 감별**: "적극적으로 참여함", "성실하게 임함" 등 반복 표현의 실질 의미 판별
7. **진정성 판단**: 과장/형식적 기록 vs 구체적 행동/사고가 드러나는 기록 구분

## 2028 대입 환경 변화 (평가 맥락)
1. **5등급제 도입**: 등급 변별력 약화 → 원점수, 평균, 표준편차, 수강자수를 종합적으로 해석해야 함
   - 등급별 비율: 1등급 10%, 2등급 24%, 3등급 32%, 4등급 24%, 5등급 10%
   - 등급별 누적 백분위: 1등급=상위 10%, 2등급=상위 34%(10+24), 3등급=상위 66%(10+24+32), 4등급=상위 90%(10+24+32+24), 5등급=하위 10%
   - **주의**: "표준편차보다 높다/낮다"는 표현 금지. 표준편차는 분산 정도를 나타내는 지표이므로 비교 대상이 아님
   - 올바른 표현 예시: "표준편차가 크므로 점수 분포가 넓어 상위 성취의 변별력이 높음", "표준편차가 작아 학생 간 점수 차이가 크지 않음"
2. 정시에서도 학생부 반영 확대 추세
3. 모집단위별 권장 이수 과목 제시 확대 → 과목 선택의 전략적 중요성 증가
4. 정성평가 강화: 수시·정시 전반에서 학교생활 충실도 평가 강화
5. 성적 인플레이션: 1등급 학생 수 증가로 동점자 경쟁 심화 → 세특 질의 차별화 중요
6. 창체 개편: 2022 개정 교육과정 학생은 3영역(자율·자치, 동아리, 진로)
7. 소인수 과목 기준: 수강자 5명 이하 시 소인수 → 성취도+등급 병기
8. **외부활동/수상경력 반영 불가**: 외부활동, 수상경력은 입시에 반영되지 않음. 생기부에도 기재 불가. AI가 외부활동을 추천하거나 언급해서는 안 됨
9. **서울대 학생부교과전형 부존재**: 서울대는 학생부교과전형을 운영하지 않음. "서울대 교과전형"을 생성하면 안 됨

## 전형 명칭 규칙 (모든 섹션에 적용)
AI가 입시 전형을 언급할 때는 반드시 아래 정식 명칭을 사용합니다:
- "학종" → **"학생부종합전형"**
- "교과" → **"학생부교과전형"**
- "정시" → **"정시(수능전형)"**
- 약칭(학종, 교과, 정시)은 프롬프트 내부 변수명에만 사용하고, 사용자에게 노출되는 텍스트에서는 정식 명칭 사용

## 외부활동 추천 금지 규칙 (모든 섹션에 적용)
외부 활동은 생기부에 입력할 수 없으므로, 다음을 준수합니다:
- 보완 전략, 주제 추천, 실행 로드맵 등 **모든 섹션에서 외부활동을 추천하지 않습니다.**
- "대회 참가", "외부 프로그램 참여", "학원 수강", "캠프 참가" 등의 추천 금지
- 모든 활동 추천은 **학교 내에서 수행 가능한 활동**으로 한정합니다.
- 추천 가능 활동 예시: 교내 수업 중 탐구, 세특 기록, 동아리 활동, 창체 활동, 교내 프로젝트

## 탐구 구조 평가 기준
탐구 활동은 다음 흐름이 있는지 확인하고, 각 단계의 존재 여부로 깊이를 판단합니다:
1. **문제 설정**: 왜 이 주제를 탐구하게 되었는지 (계기/동기)
2. **탐구 과정**: 어떤 방법으로 탐구했는지 (실험/문헌조사/비교분석/모델링 등)
3. **결과 도출**: 구체적 결과물이 있는지 (수치, 결론, 발견)
4. **확장 또는 한계 인식**: 결과에서 더 나아간 사고, 또는 한계를 인식한 흔적

| 단계 충족 | 깊이 평가 |
|-----------|----------|
| 4단계 모두 | 탐구력 우수 |
| 1~3단계 | 양호 (확장/한계 인식 부재) |
| 1~2단계만 | 미흡 (과정 있으나 결과 부족) |
| 결과만 기술 | 부족 (과정 미서술) |

## 긍정 평가 패턴
- 탐구 계기가 명확하고 주제가 구체적
- 탐구 결과가 수치화되거나 구체적 결론 도출
- 학년 간/과목 간 탐구 주제의 연속성
- 비판적 사고 (양면적 분석, 한계 인식)
- 주도적 활동 기획, 리더 역할
- 다른 과목에서 같은 역량 반복 확인 (교차 검증)

## 부정 평가 패턴
- "보고서를 작성함", "탐구를 진행함"으로 끝나고 구체적 내용 없음
- "흥미를 느낌", "유익했음", "의지를 밝힘"으로 마무리되는 추상적 결론
- 여러 활동의 깊이 없는 나열 (1500바이트에 3개 이상 활동)
- 진로와 무관한 활동 과다
- "추가 탐구 의지 표명"만 반복, 실제 추가 탐구 부재
- 학습 목표의 단순 나열 ("~를 이해함", "~를 설명함")

## 출력 규칙
- 모든 평가에는 생기부 원문에서 근거가 되는 문장이나 데이터를 반드시 인용
- 한국어로 작성하되, 전문 용어는 정확하게 사용
- 평가는 공정하고 균형 잡힌 시각 유지 (장점과 단점 모두 솔직하게)
- 구체적이고 실행 가능한 조언 제공 (추상적 권고 금지)
- **분량 기준**: 모든 평가 사유/분석 근거는 **최소 3~4줄** 이상 서술 (1~2줄의 간략한 서술 금지)
- **핵심 인용 위 분석 사유도 3~4줄 이상** 서술
- 전형 명칭은 정식 명칭 사용 (학생부종합전형, 학생부교과전형, 정시(수능전형))
- 외부활동 추천 금지 (학교 내 활동만 추천)
- "표준편차보다 높다/낮다"는 표현 금지 (표준편차는 분산 지표)
```

### 양적 콘텐츠 기준 (프롬프트에 반드시 포함)

AI 프롬프트 생성 시 아래 양적 기준을 준수하여 경쟁사 수준 이상의 콘텐츠를 생성합니다.

#### Premium 플랜 기준

- **subjectAnalysis**: 전과목(15-20과목) 분석, 과목당 sentenceAnalysis 5-10문장
- **weaknessAnalysis**: 7-10개 영역, 영역당 description 350자+, actionItems 3-5개, recordSource 매핑 필수
- **activityAnalysis**: 4개 활동유형 × 3학년 = 12개 상세 분석, 유형당 300자+ 분석, fillRate/yearlyDetails 포함
- **interviewPrep**: **최대 30문항**, 문항당 modelAnswer 200자+, followUpQuestions 2-3개, **중요도(priority: high/medium/low) 필수**
- **admissionStrategy**: 대학 추천 **총 18개** (상향 6개 + 안정 6개 + 하향 6개), 조합별 그룹핑 필수, 대학별 전형 분석 + 합격 가능성 + 구체적 준비 전략

#### Standard 플랜 기준

- **subjectAnalysis**: 10-15과목 분석, 과목당 sentenceAnalysis 3-5문장
- **weaknessAnalysis**: 5개 영역, 영역당 description 250자+
- **interviewPrep**: **최대 20문항**, **중요도(priority: high/medium/low) 필수**

#### Lite 플랜 기준

- **subjectAnalysis**: 5-8과목 분석
- **weaknessAnalysis**: 3개 영역, 영역당 description 150자+ (**최소 3~4줄**)
- **interviewPrep**: 10문항
- **admissionStrategy**: 대학 추천 **총 6개** (상향 2개 + 안정 2개 + 하향 2개)

#### 원문 인용 기준

- Premium: 과목당 최소 3개 `OriginalTextCitation`, 각 인용에 역량 태깅(`competencyTags`) 필수
- Standard: 과목당 최소 2개 원문 인용(`keyQuotes`)
- 모든 인용에 `positivePoint` 또는 `improvementSuggestion` 중 하나 이상 포함

```
학생의 강점 역량 조합에 따라 유형을 분류합니다. 유형명은 창의적이고 직관적으로 작성합니다.

유형명 예시:
- "탐구형 성장러": 학업 탐구력이 강하고 학년별 성장이 뚜렷한 학생
- "실행형 리더": 공동체 리더십이 강하고 실행력이 뛰어난 학생
- "융합형 사색가": 다양한 교과를 넘나드는 융합적 사고가 강한 학생
- "목표형 전문가": 진로 일관성이 높고 전공 분야 깊이가 있는 학생

## 역량 점수 산출 기준 (4대 역량, 각 0~100)
각 역량의 증거 품질과 양을 종합하여 0~100 점수를 부여합니다:

### 학업역량 (0~100)
- 학업성취도 (등급, 추이, 실질 위치): 40%
- 학업태도 (수업 참여, 자기주도 학습): 20%
- 탐구력 (탐구 깊이, 확장성): 40%

### 진로역량 (0~100)
- 전공 관련 교과 이수 노력: 30%
- 전공 관련 교과 성취도: 30%
- 진로 탐색 활동 및 경험: 40%

### 공동체역량 (0~100)
- 나눔과 배려: 25%
- 소통 및 협업: 25%
- 리더십: 25%
- 성실성과 규칙준수: 25%

### 발전가능성 (0~100)
- 자기주도성: 25%
- 경험의 다양성: 25%
- 성장 과정: 25%
- 창의적 문제해결: 25%

## 역량 등급 기준

| 85~100 | S (최우수) |
| 70~84 | A (우수) |

| 0~39 | D (부족) |

- 진로 관련 키워드 (예: "자동차공학", "AI윤리")

- 성장 관련 키워드 (예: "성적상승세", "활동심화")
```

```
- 합격률 수치 범위 [하한, 상한] (예: [60, 70])
- 근거 분석 (**최소 3~4줄**)

  - 정시(수능전형): 모의고사 데이터 기반 (없으면 "데이터 부족" 표시)

- 합격률 범위는 10% 단위로 설정합니다 (예: 50~60%, 70~80%).
- 합격률 산출 시 고려 요소:
  - 학생부종합전형: 역량 점수, 세특 질, 활동 일관성, 출결
  - 학생부교과전형: 내신 등급, 과목 조합, 등급 추이
  - 정시(수능전형): 모의고사 등급/백분위 (데이터 있는 경우)
- **합격률과 추천 멘트 일관성 검증**: 합격률이 높은 전형을 "보조 전략"으로, 합격률이 낮은 전형을 "주력 전략"으로 서술하는 불일치 금지
```

```
## 부족 유형 체크리스트
다음 유형의 약점이 존재하는지 확인하고, 해당되는 경우 포함하세요:

| 유형 | 설명 |
|------|------|
| 탐구 깊이 부족 | "조사함", "발표함"으로 끝나고 구체적 탐구 내용 없음 |

{competencyExtraction}
{academicAnalysis}
{studentProfile}
```

```
1. 학생의 4대 역량 프로필에 가장 적합한 전공을 선별합니다.

4. 학생이 미처 고려하지 못한 관련 전공도 제안합니다.

## 규칙
- 학생의 현재 목표 학과가 있다면 그에 대한 평가(currentTargetAssessment)를 먼저 제공합니다.
```

### 5.2 전처리: 생기부 데이터 -> 프롬프트 입력 변환

생기부는 DB에 JSONB (`record_data`)로 저장되어 있으므로, 이를 AI 프롬프트에 넣기 적합한 텍스트로 변환해야 함.

#### 5.2.1 성적 정량 계산 (코드 전처리 -- AI 호출 없음)

> **핵심**: 아래 계산은 모두 코드(`preprocessor.ts`)에서 수행. AI에게 계산을 맡기면 오류 발생 확률이 높으므로, 계산 결과만 AI에 전달하여 해석시킨다.

```typescript
interface PreprocessedAcademicData {
  // 1. 평균 등급 계산
  overallAverage: number; // 전과목 평균 등급
  averageByGrade: { year: number; semester: number; average: number }[];

  // 2. 교과 조합별 평균
  subjectCombinations: { name: string; subjects: string[]; average: number }[];

  // 3. 등급 추이
  gradeTrend: {
    direction: "ascending" | "stable" | "descending";
    magnitude: number;
  };

  // 4. Z-score + 백분위 (과목별)
  subjectStats: {
    subject: string;
    year: number;
    semester: number;
    grade: number;
    rawScore: number;
    classAverage: number;
    stdDev: number;
    zScore: number; // (rawScore - classAverage) / stdDev
    estimatedPercentile: number; // Z-score 기반 백분위 추정
    enrollmentSize: number;
  }[];

  // 5. 과목 간 편차
  gradeVariance: { highest: string; lowest: string; spread: number };

  // 6. 전공 관련 교과 성적
  majorRelated: {
    relatedAverage: number;
    overallAverage: number;
    diff: number;
  };

  // 7. 고교 유형별 등급 환산
  convertedGrade: { schoolType: string; original: number; converted: number };

  // 8. 5등급제 환산 (Premium)
  fiveGradeConversion?: {
    subject: string;
    original: number;
    converted: number;
  }[];

  // 9. 소인수 과목 식별
  smallClassSubjects: {
    subject: string;
    enrollmentSize: number;
    achievementLevel: string; // A/B/C/D/E
    grade?: string; // 등급 또는 '·'
  }[];

  // 10. 권장과목 이수 매칭
  recommendedCourseMatch: {
    targetMajor: string;
    requiredCourses: string[];
    takenCourses: string[];
    missingCourses: string[];
    matchRate: number; // 이수율 (0~1)
  };

  // 11. 교육과정 버전
  curriculumVersion: "2015" | "2022";

  // 12. 출결 데이터 정규화 (v2 추가)
  attendanceSummary: {
    year: number;
    totalAbsence: number;
    illness: number;
    unauthorized: number;
    etc: number;
    lateness: number;
    earlyLeave: number;
  }[];

  // 13. 기록 분량 계산 (v2 추가)
  recordVolume: {
    category: string; // "교과 세특", "자율활동", "동아리활동" 등
    maxCapacity: string; // "과목당 500자(1500바이트)"
    actualVolume: string; // "평균 420자"
    fillRate: number; // 0~100
  }[];

  // 14. 워드클라우드 전처리 데이터 (v2 추가)
  wordCloudData: {
    text: string;
    frequency: number;
    category?: "academic" | "career" | "community" | "growth";
  }[];

  // 15. 학생 유형 분류 입력 데이터 (v2 추가)
  studentTypeInput: {
    academicScore: number; // 학업역량 기반 점수 (전처리)
    careerScore: number; // 진로역량 기반 점수
    communityScore: number; // 공동체역량 기반 점수
    growthScore: number; // 발전가능성 기반 점수
  };
}
```

**계산 공식**:

- Z-score: `(rawScore - classAverage) / stdDev`
- 백분위 추정: 정규분포 가정, Z-score에서 CDF 변환
- 등급 환산: `require/university-grade-mapping.md`의 고교 유형별 환산 테이블 적용
- 등급 추이: 학년별 평균 등급의 선형 회귀 기울기로 방향 판단

### 5.3 에러 처리 및 재시도 전략

```
[에러 처리 계층]

Level 1: API 호출 에러
  ├─ 429 Rate Limit: 지수 백오프 (1s -> 2s -> 4s), 최대 3회
  ├─ 500 Server Error: 즉시 1회 재시도, 실패 시 5s 후 재시도, 최대 3회

  ├─ JSON 파싱 실패: 재시도 1회 (temperature 0.1로 낮춤)

Level 4: 섹션 실패 (3회 재시도 후에도 실패)
```

### 5.4 비동기 처리 아키텍처

리포트 생성은 결제 후 비동기로 처리. 48시간 내 배달 목표이지만, 실제 생성 시간은 2~5분 예상.

**아키텍처 옵션 (권장: Option A)**:

**Option A: Supabase Edge Function + Queue (권장)**

```
결제 완료
  -> reports 테이블에 status='pending' 레코드 생성
  -> Edge Function 트리거 (혹은 pg_notify -> webhook)
  -> Edge Function에서 Gemini API 순차/병렬 호출
  -> 완료 시 reports.content 업데이트 + status='completed'
  -> 사용자 알림 발송
```

**Option B: Next.js API Route + Background Job**

```
결제 완료
  -> API Route에서 reports 레코드 생성
  -> Inngest/Trigger.dev 등 Job Queue에 작업 등록
  -> Worker에서 Gemini API 호출
  -> 완료 시 DB 업데이트 + 알림
```

**권장 이유 (Option A)**:

- 이미 Supabase를 사용 중이므로 인프라 추가 없음
- Edge Function은 최대 400초(~6.5분) 실행 가능
- 별도 Job Queue 인프라 불필요

**상태 관리**:

```
reports.status: 'pending' -> 'processing' -> 'completed' | 'failed'
reports.progress: 0~100 (섹션 완료 비율)
reports.current_section: '현재 처리 중인 섹션명' (선택적, UX용)
```

### 7.2 할루시네이션 방지

1. **원문 인용 강제**: 시스템 프롬프트에서 "모든 평가에는 생기부 원문 인용 필수" 명시
2. **원문 대조 검증** (후처리): AI가 인용한 문장이 실제 생기부에 존재하는지 코드로 교차 검증
   - 완전 일치가 아닌, 유사도 기반 매칭 (생기부 원문의 부분 인용 허용)
   - 매칭 실패 시 경고 로그 기록 (초기에는 차단하지 않고 모니터링)
3. **대학 추천 범위 제한**: 대학 등급 매핑 테이블에 있는 대학만 추천하도록 프롬프트에 명시
4. **성적 데이터 정합성**: AI가 출력한 등급/점수가 입력 데이터와 일치하는지 코드 검증
5. **도서 추천 검증**: AI가 추천한 도서가 실재하는지 외부 API로 교차 검증 (선택적)

### 7.3 평가 일관성 보장

1. **낮은 Temperature**: 모든 호출에 `temperature: 0.3` 적용 (일관된 평가 유도)
2. **Few-shot 예시**: 핵심 섹션(세특 분석, 역량 평가)에 1~2개 예시를 프롬프트에 포함
   - 예시는 예시 분석본에서 추출한 실제 평가 패턴 사용
3. **명확한 등급 기준**: S/A/B/C/D, excellent/good/average/weak 등급의 구체적 기준을 프롬프트에 명시
4. **Phase 2 선행 분석**: 역량 추출과 성적 분석을 선행하여, 이후 섹션들이 동일한 분석 결과를 참조하므로 섹션 간 모순 방지
5. **학생 유형 분류 선행**: Call C에서 산출된 점수/등급이 이후 섹션의 기준점 역할

### 7.4 Few-shot 예시 전략

모든 섹션에 few-shot 예시를 넣으면 입력 토큰이 과도하게 증가. 다음 전략으로 선별 적용:

**Few-shot 필수 섹션** (평가 품질에 직접 영향):

- 세특 분석 (subjectAnalysis): 좋은 평가 예시 1개 + 나쁜 평가 예시 1개
- 역량별 종합 평가 (competencyEvaluation): 장점/단점 서술 예시 각 1개
- Premium 문장 단위 분석: 문장 분석 예시 2개 (긍정 1개 + 부정 1개)

**Few-shot 불필요 섹션** (지시문만으로 충분):

- 종합 진단: 한줄 총평 예시를 지시문에 내장
- 학생 프로필: 유형명/캐치프레이즈 예시를 지시문에 내장
- 면접 질문: 질문 형식이 명확하므로 지시문으로 충분
- 실행 로드맵: 구조화된 출력 스키마가 형식을 제어
- 워드 클라우드: 코드 전처리이므로 해당 없음

**예시 관리**: 별도 프롬프트 모듈(`libs/report/prompts/examples/`)에 저장하여, 프롬프트 빌더가 필요 시 동적으로 조합.
